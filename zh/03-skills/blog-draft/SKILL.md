## 使用者輸入

```text
$ARGUMENTS
```

在繼續之前，你**必須**先考慮使用者輸入。使用者應提供：
- **想法/主題**：部落格文章的核心概念或主題
- **資源**：用於研究的 URL、檔案或參考資料（可選，但推薦）
- **目標讀者**：這篇部落格是寫給誰看的（可選）
- **語氣/風格**：正式、輕鬆、技術向等（可選）

**重要**：如果使用者是在請求更新一篇**已有部落格文章**，請跳過步驟 0-8，直接從**步驟 9** 開始。先閱讀現有草稿檔案，再繼續迭代流程。

## 執行流程

請按順序執行以下步驟。**不要跳步，也不要在未獲得使用者批准的情況下繼續執行標明需要確認的步驟。**

### 步驟 0：建立專案資料夾

1. 使用以下格式生成資料夾名稱：`YYYY-MM-DD-short-topic-name`
   - 使用今天的日期
   - 根據主題生成一個簡短、適合 URL 的 slug（小寫、連字元連線，最多 5 個詞）

2. 建立資料夾結構：
   ```text
   blog-posts/
   └── YYYY-MM-DD-short-topic-name/
       └── resources/
   ```

3. 在繼續前，先向使用者確認資料夾已建立。

### 步驟 1：研究與資源收集

1. 在部落格文章目錄中建立 `resources/` 子資料夾

2. 對於每個提供的資源：
   - **URL**：抓取關鍵內容並儲存到 `resources/` 下的 markdown 檔案中
   - **檔案**：讀取並在 `resources/` 中做摘要
   - **主題**：使用網路搜尋收集最新資訊

3. 為每個資源在 `resources/` 中建立摘要檔案：
   - `resources/source-1-[short-name].md`
   - `resources/source-2-[short-name].md`
   - 以此類推

4. 每個摘要都應包含：
   ```markdown
   # 來源：[標題/URL]

   ## 要點
   - 要點 1
   - 要點 2

   ## 相關引文/資料
   - 引文或統計 1
   - 引文或統計 2

   ## 與主題的關聯
   簡要說明其相關性
   ```

5. 向使用者展示研究摘要。

### 步驟 2：頭腦風暴與澄清

1. 基於想法和已研究的資源，展示：
   - 從研究中識別出的**主要主題**
   - 部落格文章的**可能切入角度**
   - 應該覆蓋的**關鍵點**
   - 仍需進一步澄清的**資訊缺口**

2. 提出澄清問題：
   - 你希望讀者最終帶走的核心結論是什麼？
   - 研究中有沒有哪些點是你希望重點強調的？
   - 目標長度是多少？（短：500-800 字，中：1000-1500 字，長：2000+ 字）
   - 有沒有哪些內容你希望排除？

3. **在繼續之前，等待使用者回覆。**

### 步驟 3：提出提綱

1. 建立一個結構化提綱，包括：

   ```markdown
   # 部落格文章提綱：[標題]

   ## 元資訊
   - **目標讀者**：[誰]
   - **語氣**：[風格]
   - **目標長度**：[字數]
   - **核心結論**：[關鍵資訊]

   ## 建議結構

   ### 開場/引子
   - 開場鉤子思路
   - 背景鋪墊
   - 論點陳述

   ### 第一部分：[標題]
   - 關鍵點 A
   - 關鍵點 B
   - 來自 [來源] 的支撐證據

   ### 第二部分：[標題]
   - 關鍵點 A
   - 關鍵點 B

   [繼續列出所有部分...]

   ### 結論
   - 關鍵點總結
   - 行動號召或最終思考

   ## 需要引用的來源
   - 來源 1
   - 來源 2
   ```

2. 向使用者展示提綱，並**請求批准或修改意見**。

### 步驟 4：儲存已批准的提綱

1. 使用者批准提綱後，將其儲存為部落格文章資料夾中的 `OUTLINE.md`。

2. 確認提綱已儲存。

### 步驟 5：提交提綱（如果在 git 倉庫中）

1. 檢查當前目錄是否為 git 倉庫。

2. 如果是：
   - 暫存新檔案：部落格文章資料夾、resources 和 `OUTLINE.md`
   - 使用如下提交資訊建立 commit：`docs: Add outline for blog post - [topic-name]`
   - 推送到遠端倉庫

3. 如果不是 git 倉庫，則跳過此步驟並告知使用者。

### 步驟 6：撰寫草稿

1. 基於已批准的提綱，撰寫完整部落格草稿。

2. 嚴格按照 `OUTLINE.md` 的結構進行編寫。

3. 包含：
   - 有吸引力的引言和開場鉤子
   - 清晰的章節標題
   - 來自研究的支撐證據和示例
   - 段落之間自然流暢的過渡
   - 強有力的結尾和核心收穫
   - **引用**：所有比較、統計資料和事實性陳述都必須引用原始來源

4. 將草稿儲存為部落格文章資料夾中的 `draft-v0.1.md`。

5. 格式：
   ```markdown
   # [部落格文章標題]

   *[可選：副標題或標語]*

   [包含行內引用的完整正文...]

   ---

   ## 參考資料
   - [1] 來源 1 標題 - URL 或引用
   - [2] 來源 2 標題 - URL 或引用
   - [3] 來源 3 標題 - URL 或引用
   ```

6. **引用要求**：
   - 每個資料點、統計資訊或比較都必須帶有行內引用
   - 使用編號引用 [1]、[2] 等，或命名引用 [Source Name]
   - 在文末的參考資料部分連結這些引用
   - 示例：“研究表明，65% 的開發者更偏好 TypeScript [1]”
   - 示例：“React 在渲染速度上比 Vue 快 20% [React Benchmarks 2024]”

### 步驟 7：提交草稿（如果在 git 倉庫中）

1. 檢查是否處於 git 倉庫中。

2. 如果是：
   - 暫存草稿檔案
   - 使用如下提交資訊建立 commit：`docs: Add draft v0.1 for blog post - [topic-name]`
   - 推送到遠端倉庫

3. 如果不是 git 倉庫，則跳過並告知使用者。

### 步驟 8：向使用者展示草稿以供審閱

1. 向使用者展示草稿內容。

2. 詢問反饋：
   - 整體印象如何？
   - 哪些部分需要擴充套件或壓縮？
   - 是否需要調整語氣？
   - 是否缺少關鍵資訊？
   - 有沒有具體編輯或重寫建議？

3. **等待使用者回覆。**

### 步驟 9：迭代或定稿

**如果使用者要求修改：**
1. 記錄所有修改請求
2. 返回步驟 6，並做以下調整：
   - 將版本號遞增（v0.2、v0.3 等）
   - 納入所有反饋
   - 儲存為 `draft-v[X.Y].md`
   - 重複步驟 7-8

**如果使用者批准：**
1. 確認最終草稿版本
2. 如使用者要求，可重新命名為 `final.md`
3. 總結部落格文章建立過程：
   - 一共建立了多少個版本
   - 各版本之間的關鍵變化
   - 最終字數
   - 建立了哪些檔案

## 版本跟蹤

所有草稿都會以遞增版本號保留：
- `draft-v0.1.md` - 初始草稿
- `draft-v0.2.md` - 第一輪反饋後
- `draft-v0.3.md` - 第二輪反饋後
- 以此類推

這可以跟蹤部落格文章的演變過程，並在需要時回退。

## 輸出檔案結構

```text
blog-posts/
└── YYYY-MM-DD-topic-name/
    ├── resources/
    │   ├── source-1-name.md
    │   ├── source-2-name.md
    │   └── ...
    ├── OUTLINE.md
    ├── draft-v0.1.md
    ├── draft-v0.2.md（如果有迭代）
    └── draft-v0.3.md（如果還有更多迭代）
```

## 質量建議

- **鉤子**：以問題、令人驚訝的事實或貼近讀者的場景開頭
- **銜接**：每一段都要自然連線到下一段
- **證據**：用研究資料支撐觀點
- **引用**：以下內容務必引用來源：
  - 所有統計資料和數值（例如：“根據 [Source]，75% 的……”）
  - 對產品、服務或方案的比較（例如：“X 的速度比 Y 快 2 倍 [Source]”）
  - 關於市場趨勢、研究發現或基準測試的事實性陳述
  - 使用行內引用，格式為：[Source Name] 或 [Author, Year]
- **語氣**：全程保持一致
- **長度**：尊重目標字數
- **可讀性**：使用短段落，並在適當位置使用專案符號
- **CTA**：以明確的行動號召或發人深省的問題收尾

## 備註

- 始終在上述需要確認的節點等待使用者批准
- 保留所有草稿版本，方便追蹤歷史
- 如果提供了 URL，使用網路搜尋獲取最新資訊
- 如果資源不足，詢問使用者補充，或建議進一步研究
- 根據目標讀者調整語氣（技術、通用、商業等）
