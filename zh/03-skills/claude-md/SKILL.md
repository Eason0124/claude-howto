## 使用者輸入

```text
$ARGUMENTS
```

在繼續之前，你**必須**先考慮使用者輸入（如果不為空）。使用者可能會指定：
- `create` - 從零建立新的 CLAUDE.md
- `update` - 改進已有的 CLAUDE.md
- `audit` - 分析並報告當前 CLAUDE.md 的質量
- 一個具體路徑，用於建立或更新（例如 `src/api/CLAUDE.md` 代表目錄級說明）

## 核心原則

**LLM 是無狀態的**：CLAUDE.md 是每次對話中唯一會自動包含的檔案。它是讓 AI agent 瞭解程式碼庫的主要入門檔案。

### 黃金法則

1. **少即是多**：前沿 LLM 大約能遵循 150-200 條指令。Claude Code 的系統提示詞本身已經佔了大約 50 條，因此 CLAUDE.md 必須聚焦且簡潔。

2. **只放通用資訊**：只包含每次會話都適用的內容。任務特定的說明應該放在單獨檔案裡。

3. **不要把 Claude 當成 lint 工具**：風格指南會膨脹上下文並降低指令遵循效果。應改用確定性工具（如 prettier、eslint 等）。

4. **絕不自動生成**：CLAUDE.md 是 AI harness 中槓桿最高的位置。應該經過認真思考後手工編寫。

## 執行流程

### 1. 專案分析

首先分析當前專案狀態：

1. 檢查是否存在已有的 CLAUDE.md 檔案：
   - 根目錄：`./CLAUDE.md` 或 `.claude/CLAUDE.md`
   - 目錄級：`**/CLAUDE.md`
   - 全域性使用者配置：`~/.claude/CLAUDE.md`

2. 識別專案結構：
   - 技術棧（語言、框架）
   - 專案型別（monorepo、單體應用、庫）
   - 開發工具（包管理器、構建系統、測試執行器）

3. 檢視已有檔案：
   - README.md
   - CONTRIBUTING.md
   - package.json、pyproject.toml、Cargo.toml 等

### 2. 內容策略（WHAT, WHY, HOW）

圍繞三個維度組織 CLAUDE.md：

#### WHAT - 技術與結構
- 技術棧概覽
- 專案組織方式（對 monorepo 尤其重要）
- 關鍵目錄及其用途

#### WHY - 目的與背景
- 專案是做什麼的
- 為什麼做出這些架構決策
- 每個主要元件負責什麼

#### HOW - 工作流與約定
- 開發流程（bun vs node、pip vs uv 等）
- 測試流程和命令
- 驗證與構建方法
- 關鍵“坑點”或非顯而易見的要求

### 3. 漸進式披露策略

對於較大的專案，建議建立 `agent_docs/` 資料夾：

```text
agent_docs/
  |- building_the_project.md
  |- running_tests.md
  |- code_conventions.md
  |- architecture_decisions.md
```

在 CLAUDE.md 中引用這些檔案，並寫明：
```markdown
關於詳細的構建說明，請參考 `agent_docs/building_the_project.md`
```

**重要**：使用 `file:line` 引用，而不是程式碼片段，以避免上下文過時。

### 4. 質量約束

建立或更新 CLAUDE.md 時：

1. **目標長度**：少於 300 行，理想情況下少於 100 行
2. **不要寫風格規則**：移除任何 lint/format 相關說明
3. **不要寫任務特定說明**：移到單獨檔案中
4. **不要寫程式碼片段**：改用檔案引用
5. **不要重複資訊**：不要重複 package.json 或 README 中已有內容

### 5. 必備章節

一個結構良好的 CLAUDE.md 應包含：

```markdown
# 專案名稱

一句簡短的專案描述。

## 技術棧
- 主語言和版本
- 關鍵框架/庫
- 資料庫/儲存（如有）

## 專案結構
[僅適用於 monorepo 或複雜結構]
- `apps/` - 應用入口
- `packages/` - 共享庫

## 開發命令
- 安裝：`command`
- 測試：`command`
- 構建：`command`

## 關鍵約定
[只保留非顯而易見、高影響的約定]
- 約定 1，簡要說明
- 約定 2，簡要說明

## 已知問題 / 坑點
[經常讓開發者踩坑的內容]
- 問題 1
- 問題 2
```

### 6. 需要避免的反模式

**不要包含：**
- 程式碼風格指南（交給 lint 工具）
- 如何使用 Claude 的說明
- 對顯而易見模式的冗長解釋
- 複製貼上的程式碼示例
- 泛泛而談的最佳實踐（例如“寫乾淨的程式碼”）
- 特定任務的說明
- 自動生成內容
- 大量 TODO 列表

### 7. 驗證清單

在最終確定前，檢查：

- [ ] 少於 300 行（最好少於 100 行）
- [ ] 每一行都適用於所有會話
- [ ] 沒有風格/格式規則
- [ ] 沒有程式碼片段（使用檔案引用）
- [ ] 命令已經驗證可用
- [ ] 對複雜專案使用了漸進式披露
- [ ] 已記錄關鍵坑點
- [ ] 沒有與 README.md 重複

## 輸出格式

### 對於 `create` 或預設模式：

1. 分析專案
2. 按上述結構起草 CLAUDE.md
3. 向使用者展示草稿以供審閱
4. 在獲得批准後寫入相應位置

### 對於 `update`：

1. 讀取現有 CLAUDE.md
2. 按最佳實踐進行審計
3. 識別：
   - 需要刪除的內容（風格規則、程式碼片段、任務特定內容）
   - 需要壓縮的內容
   - 缺失的重要資訊
4. 向使用者展示修改建議
5. 在獲得批准後應用修改

### 對於 `audit`：

1. 讀取現有 CLAUDE.md
2. 生成報告，包括：
   - 當前行數與目標的對比
   - 可通用於所有會話內容的百分比
   - 發現的反模式列表
   - 改進建議
3. 不要修改檔案，只做報告

## AGENTS.md 處理

如果使用者請求建立或更新 AGENTS.md：

AGENTS.md 用於定義專門的 agent 行為。與 CLAUDE.md（專案上下文）不同，AGENTS.md 定義的是：
- 自定義 agent 角色和能力
- agent 級約束與說明
- 多 agent 場景下的工作流定義

同樣適用以下原則：
- 保持聚焦和簡潔
- 使用漸進式披露
- 用外部檔案引用代替內聯內容

## 備註

- 在寫入前始終驗證命令是否可用
- 不確定時就不要寫，少即是多
- 系統提醒會告訴 Claude，CLAUDE.md “可能相關，也可能不相關” - 噪音越多，越容易被忽略
- monorepo 最需要清晰的 WHAT/WHY/HOW 結構
- 目錄級 CLAUDE.md 應該更聚焦
