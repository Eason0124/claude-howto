# 程式碼重構 Skill

這是一個基於 Martin Fowler《Refactoring: Improving the Design of Existing Code》（第 2 版）的系統化程式碼重構方法。這個 skill 強調安全、漸進的修改，並由測試提供保障。

> “重構是在不改變軟體外部行為的前提下，改進其內部結構的過程。” — Martin Fowler

## 核心原則

1. **保持行為不變**：外部行為必須保持一致
2. **小步前進**：每次只做很小、可測試的改動
3. **測試驅動**：測試是安全網
4. **持續進行**：重構是長期過程，不是一次性任務
5. **協作確認**：每個階段都需要使用者確認

## 工作流程總覽

```
階段 1：研究與分析
    ↓
階段 2：測試覆蓋評估
    ↓
階段 3：識別程式碼異味
    ↓
階段 4：建立重構計劃
    ↓
階段 5：增量實施
    ↓
階段 6：評審與迭代
```

---

## 階段 1：研究與分析

### 目標
- 理解程式碼庫結構和用途
- 確定重構範圍
- 收集業務需求背景

### 先向使用者確認的問題
開始前先確認：

1. **範圍**：哪些檔案 / 模組 / 函式需要重構？
2. **目標**：你想解決什麼問題？（可讀性、效能、可維護性）
3. **約束**：有哪些區域不能改？
4. **時間壓力**：這是否阻塞了其他工作？
5. **測試狀態**：是否已有測試？是否透過？

### 行動
- [ ] 閱讀並理解目的碼
- [ ] 識別依賴和整合點
- [ ] 記錄當前架構
- [ ] 標記已有技術債跡象（TODO、FIXME）

### 輸出
向使用者彙報：
- 程式碼結構總結
- 識別出的問題區域
- 初步建議
- **請求繼續執行的批准**

## 階段 2：測試覆蓋評估

### 為什麼測試重要
> “沒有測試的重構，就像沒有安全帶就開車。” — Martin Fowler

測試是安全重構的**關鍵前提**。沒有測試，很容易引入 bug。

### 評估步驟

1. **檢查現有測試**
   ```bash
   # 查詢測試檔案
   find . -name "*test*" -o -name "*spec*" | head -20
   ```

2. **執行現有測試**
   ```bash
   # JavaScript / TypeScript
   npm test

   # Python
   pytest -v

   # Java
   mvn test
   ```

3. **檢查覆蓋率（如果可用）**
   ```bash
   # JavaScript
   npm run test:coverage

   # Python
   pytest --cov=.
   ```

### 決策點：詢問使用者

**如果測試存在且透過：**
- 進入階段 3

**如果測試缺失或不完整：**
給出選項：
1. 先寫測試（推薦）
2. 在重構過程中逐步補測試
3. 不寫測試直接繼續（有風險，需要使用者確認）

**如果測試失敗：**
- 停止。先修復失敗測試再重構
- 問使用者：是否先修測試？

### 測試編寫建議（如需要）

對每個要重構的函式，測試應覆蓋：
- 正常路徑
- 邊界情況（空輸入、null、邊界值）
- 錯誤場景（非法輸入、異常）

使用“紅-綠-重構”迴圈：
1. 寫失敗測試（紅）
2. 讓它透過（綠）
3. 再重構

## 階段 3：識別程式碼異味

### 什麼是程式碼異味？
程式碼深層問題的表面症狀。它們不一定是 bug，但說明程式碼設計可能有問題。

### 常見異味清單

完整目錄見 [references/code-smells.md](references/code-smells.md)。

#### 快速參考

| 異味 | 跡象 | 影響 |
|-------|------|------|
| **長函式** | 函式超過 30-50 行 | 難以理解、測試和維護 |
| **重複程式碼** | 多處出現相同邏輯 | 修復需要改多處 |
| **大類** | 類承擔了太多職責 | 違反單一職責原則 |
| **Feature Envy** | 一個方法更多依賴別的類的資料 | 封裝性差 |
| **基礎型別沉迷** | 過度使用基礎型別而不是物件 | 缺少領域概念 |
| **長引數列表** | 方法引數超過 4 個 | 呼叫困難 |
| **資料泥團** | 一組資料總是一起出現 | 缺少抽象 |
| **switch 語句** | 複雜的 switch / if-else 鏈 | 難以擴充套件 |
| **臆想泛化** | “以防未來需要”提前設計 | 不必要的複雜度 |
| **死程式碼** | 未使用的程式碼 | 造成困惑和維護負擔 |

### 分析步驟

1. **自動分析**（如果有指令碼）
   ```bash
   python scripts/detect-smells.py <file>
   ```

2. **人工審查**
   - 系統地走讀程式碼
   - 記錄每個異味的位置和嚴重性
   - 按影響分類（Critical / High / Medium / Low）

3. **優先順序排序**
   優先關注會：
   - 阻塞當前開發的異味
   - 導致 bug 或混淆的異味
   - 影響最常變更程式碼路徑的異味

### 輸出：異味報告

向使用者呈現：
- 識別出的異味及位置
- 每項的嚴重性評估
- 建議的優先順序順序
- **請求使用者確認優先順序**

## 階段 4：建立重構計劃

### 選擇重構方式

針對每個異味，從目錄中選擇合適的重構手法。

完整列表見 [references/refactoring-catalog.md](references/refactoring-catalog.md)。

#### 異味到重構的對映

| 程式碼異味 | 推薦重構 |
|----------|----------|
| 長函式 | Extract Method、Replace Temp with Query |
| 重複程式碼 | Extract Method、Pull Up Method、Form Template Method |
| 大類 | Extract Class、Extract Subclass |
| Feature Envy | Move Method、Move Field |
| 基礎型別沉迷 | Replace Primitive with Object、Replace Type Code with Class |
| 長引數列表 | Introduce Parameter Object、Preserve Whole Object |
| 資料泥團 | Extract Class、Introduce Parameter Object |
| switch 語句 | Replace Conditional with Polymorphism |
| 臆想泛化 | Collapse Hierarchy、Inline Class、Remove Dead Code |
| 死程式碼 | Remove Dead Code |

### 計劃結構

使用 [templates/refactoring-plan.md](templates/refactoring-plan.md) 裡的模板。

每項重構都要寫明：
1. **Target**：將修改哪些程式碼
2. **Smell**：解決什麼問題
3. **Refactoring**：採用哪種手法
4. **Steps**：詳細微步驟
5. **Risks**：可能出什麼問題
6. **Rollback**：如何回退

### 分階段方法

**關鍵**：重構必須分階段推進。

**階段 A：快速收益**（低風險，高價值）
- 重新命名變數以提升清晰度
- 提取明顯重複的程式碼
- 刪除死程式碼

**階段 B：結構改進**（中風險）
- 從長函式中提取方法
- 引入引數物件
- 把方法移動到更合適的類

**階段 C：架構改動**（高風險）
- 用多型替代條件分支
- 提取類
- 引入設計模式

### 決策點：把計劃展示給使用者

在開始實施前：
- 展示完整重構計劃
- 解釋每個階段及其風險
- 獲得每個階段的明確批准
- **詢問**：“是否繼續執行階段 A？”

## 階段 5：增量實施

### 黃金法則
> “修改 → 測試 → 透過？→ 提交 → 下一步”

### 實施節奏

對每一步重構：

1. **預檢查**
   - 測試透過（綠色）
   - 程式碼能編譯

2. **只做一個小改動**
   - 按目錄中的具體操作進行
   - 保持改動最小化

3. **驗證**
   - 立刻執行測試
   - 檢查編譯錯誤

4. **如果測試透過（綠色）**
   - 用描述清晰的提交資訊提交
   - 繼續下一步

5. **如果測試失敗（紅色）**
   - 立刻停止
   - 撤銷改動
   - 分析原因
   - 如有疑問，詢問使用者

### 提交策略

每次提交都應當：
- **原子性**：只包含一個邏輯改動
- **可回滾**：容易撤銷
- **描述清楚**：提交資訊明確

示例提交資訊：
```
refactor: 從 processOrder() 中提取 calculateTotal()
refactor: 將 'x' 重新命名為 'customerCount' 以提升清晰度
refactor: 刪除未使用的 validateOldFormat() 方法
```

### 進度彙報

每個子階段完成後，向使用者彙報：
- 做了哪些改動
- 測試是否仍透過
- 遇到了什麼問題
- **詢問**：“繼續下一批嗎？”

## 階段 6：評審與迭代

### 重構後檢查清單

- [ ] 所有測試透過
- [ ] 沒有新的警告 / 錯誤
- [ ] 程式碼編譯成功
- [ ] 行為沒有變化（手動驗證）
- [ ] 必要時已更新檔案
- [ ] 提交歷史乾淨

### 指標對比

重構前後執行複雜度分析：
```bash
python scripts/analyze-complexity.py <file>
```

展示改進：
- 程式碼行數變化
- 圈複雜度變化
- 可維護性指標變化

### 使用者評審

向使用者展示最終結果：
- 所有變更摘要
- 重構前後程式碼對比
- 指標改善情況
- 剩餘技術債
- **詢問**：“你對這些改動滿意嗎？”

### 下一步

和使用者討論：
- 還要處理哪些異味？
- 是否安排下一次重構？
- 是否把類似修改應用到其他地方？

## 重要指南

### 何時暫停並詢問

遇到以下情況時，務必暫停並和使用者確認：
- 不確定業務邏輯
- 改動可能影響外部 API
- 測試覆蓋不足
- 需要做重大的架構決策
- 風險上升
- 遇到意外複雜性

### 安全規則

1. **沒有測試不要重構**（除非使用者明確確認風險）
2. **不要做大改動**，拆成小步
3. **每次改動後都不要跳過測試**
4. **測試失敗就不要繼續**，先修復或回滾
5. **不要臆測**，不確定就問

### 不要做什麼

- 不要把重構和新功能混在一起
- 不要在生產事故期間做重構
- 不要重構你看不懂的程式碼
- 不要過度設計，保持簡單
- 不要一次性重構所有內容

## 快速上手示例

### 場景：長函式 + 重複邏輯

**重構前：**
```javascript
function processOrder(order) {
  // 150 行程式碼，包含：
  // - 重複驗證邏輯
  // - 內聯計算
  // - 多種職責混雜
}
```

**重構步驟：**

1. **確認測試存在**，覆蓋 `processOrder()`
2. **提取** 驗證邏輯為 `validateOrder()`
3. **測試** - 應該透過
4. **提取** 計算邏輯為 `calculateOrderTotal()`
5. **測試** - 應該透過
6. **提取** 通知邏輯為 `notifyCustomer()`
7. **測試** - 應該透過
8. **評審** - `processOrder()` 現在只負責串聯 3 個清晰函式

**重構後：**
```javascript
function processOrder(order) {
  validateOrder(order);
  const total = calculateOrderTotal(order);
  notifyCustomer(order, total);
  return { order, total };
}
```
