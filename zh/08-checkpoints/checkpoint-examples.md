# 檢查點示例

以下是一些在 Claude Code 中有效使用檢查點的真實示例。

注意：每次使用者提示都會自動建立檢查點。你不需要手動儲存。要回退，請連續按兩次 `Esc`（`Esc+Esc`），或者使用 `/rewind` 開啟檢查點瀏覽器。

## 示例 1：資料庫遷移

### 場景
你需要從 MySQL 遷移到 PostgreSQL，但想同時探索直接遷移和漸進式雙寫兩種方案。

### 工作流程

```
使用者：我們來實現一個從 MySQL 到 PostgreSQL 的直接遷移方案
      [檢查點 A 自動建立]

Claude：我來實現直接遷移……
[更換資料庫驅動、更新查詢、修改 schema]

使用者：執行測試
      [檢查點 B 自動建立]

Claude：正在執行測試……
❌ 15 個測試失敗 - 連線池問題

使用者：這個方案有問題。我們改試雙寫模式。
[使用者按下 Esc+Esc，選擇檢查點 A，選擇“恢復程式碼和對話”]

使用者：現在實現雙寫模式，同時向兩個資料庫寫入

Claude：我來實現雙寫遷移……
[實現雙寫層、漸進式遷移策略]

使用者：執行測試

Claude：正在執行測試……
✅ 所有測試透過

使用者：雙寫方案明顯更穩妥。我們提交這個版本。
```

## 示例 2：效能最佳化

```
使用者：我想試試另一種方法。
[使用者按下 Esc+Esc，選擇“Baseline”檢查點，選擇“恢復程式碼和對話”]

使用者：改試資料庫查詢最佳化

Claude：我來最佳化資料庫查詢……
[新增索引，最佳化 N+1 查詢]

使用者：執行效能測試

Claude：結果：平均響應時間 180ms（提升 60%）

使用者：我也想試第三種方法。
[使用者按下 Esc+Esc，選擇“Baseline”檢查點，選擇“恢復程式碼和對話”]

使用者：嘗試響應壓縮和 CDN

Claude：我來新增壓縮和 CDN……
[實現 gzip，配置 CDN]

使用者：執行效能測試

Claude：結果：平均響應時間 320ms（提升 29%）

使用者：查詢最佳化最好。讓我回到那個版本，再和快取結合起來。
[使用者按下 Esc+Esc，選擇查詢最佳化後的檢查點，選擇“恢復程式碼和對話”]

使用者：現在在這些最佳化後的查詢之上再加上 Redis 快取

Claude：[在最佳化後的查詢之上新增 Redis 快取]

使用者：執行效能測試

Claude：結果：平均響應時間 95ms（提升 79%）🎉

使用者：完美！這就是最終方案。我們提交它。
```

## 示例 4：除錯會話

### 場景
用多個假設排查一個難纏的 bug。

### 工作流程

```
使用者：應用裡有記憶體洩漏
      [檢查點“Before debugging”自動建立]

Claude：讓我來調查……
[分析程式碼]

使用者：我覺得問題在事件監聽器裡。我們先修這個。

Claude：[修復事件監聽器清理邏輯]

使用者：執行記憶體分析器

Claude：仍然在洩漏記憶體 - 事件監聽器不是問題所在

使用者：讓我回到之前，換個思路試試。
[使用者按下 Esc+Esc，選擇“Before debugging”檢查點，選擇“恢復程式碼和對話”]

使用者：也許是資料庫連線的問題？

Claude：[實現連線池修復]

使用者：執行記憶體分析器

Claude：還是在洩漏 - 也不是連線的問題

使用者：我再回去，換另一個方向試試。
[使用者按下 Esc+Esc，選擇“Before debugging”檢查點，選擇“恢復程式碼和對話”]

使用者：我們檢查一下有沒有迴圈引用

Claude：[找到並修復快取中的迴圈引用]

使用者：執行記憶體分析器

Claude：✅ 記憶體洩漏修復了！快取層裡的迴圈引用就是罪魁禍首。

使用者：太好了！我們提交這個修復。
```

## 示例 8：從檢查點開始總結

### 場景
在一次很長的除錯會話後，你想壓縮對話內容，同時保留上下文。

### 工作流程

```
使用者：[經過 20 多條訊息的除錯和探索之後]

[使用者按下 Esc+Esc，選擇較早的檢查點，選擇“從這裡開始總結”]
[可選地提供指令：“重點保留我們嘗試過什麼，以及什麼有效”]

Claude：[生成從該點開始的對話總結]
[原始訊息保留在記錄裡]
[總結會替換當前可見對話，從而減少上下文視窗佔用]

使用者：現在繼續推進那個有效的方案。
```

## 關鍵結論

1. **檢查點是自動建立的**：每次使用者提示都會建立一個檢查點，不需要手動儲存。
2. **使用 `Esc+Esc` 或 `/rewind`**：這兩種方式都可以開啟檢查點瀏覽器。
3. **選擇合適的恢復方式**：根據需要恢復程式碼、對話、兩者都恢復，或者直接總結。
4. **不要害怕試驗**：檢查點讓你可以安全嘗試激進的改動。
5. **和 git 結合使用**：檢查點用於探索，git 用於最終落地。
6. **長會話要及時總結**：用“從這裡開始總結”來保持對話可管理。
