# 安全政策

## 概述

Claude How To 專案的安全對我們非常重要。本檔案說明我們的安全實踐，以及如何負責任地報告安全漏洞。

## 支援的版本

我們為以下版本提供安全更新：

| 版本 | 狀態 | 支援至 |
|---------|--------|---------------|
| 最新版（main） | ✅ 活躍 | 當前版本 + 6 個月 |
| 1.x 釋出版 | ✅ 活躍 | 直到下一個大版本 |

**注意**：作為一個教育指南專案，我們更關注維護當前最佳實踐和檔案安全，而不是傳統意義上的版本支援。更新會直接應用到 main 分支。

## 安全實踐

### 程式碼安全

1. **依賴管理**
   - 所有 Python 依賴都固定在 `requirements.txt` 中
   - 透過 dependabot 和人工審查進行定期更新
   - 每次提交都會執行 Bandit 安全掃描
   - 使用 pre-commit hooks 做安全檢查

2. **程式碼質量**
   - 使用 Ruff 做 lint，能捕獲常見問題
   - 使用 mypy 做型別檢查，避免型別相關漏洞
   - 透過 pre-commit hooks 強制執行標準
   - 所有更改在合併前都要審查

3. **訪問控制**
   - 對 `main` 分支啟用分支保護
   - 合併前需要審查透過
   - 狀態檢查必須透過後才能合併
   - 倉庫寫許可權受限

### 檔案安全

1. **示例中不放金鑰**
   - 所有示例中的 API key 都使用佔位符
   - 從不硬編碼憑據
   - `.env.example` 檔案會展示所需變數
   - 明確提示金鑰管理風險

2. **安全最佳實踐**
   - 示例會展示安全模式
   - 檔案中會突出安全警告
   - 連結到官方安全指南
   - 在相關章節中說明憑據處理方式

3. **內容審查**
   - 所有檔案都會審查安全問題
   - 貢獻指南中包含安全注意事項
   - 會驗證外部連結和引用

### 依賴安全

1. **掃描**
   - Bandit 會掃描所有 Python 程式碼中的漏洞
   - 透過 GitHub 安全警報檢查依賴漏洞
   - 定期進行人工安全審計

2. **更新**
   - 安全補丁會盡快應用
   - 大版本會謹慎評估
   - 更新日誌會包含安全相關變更

3. **透明度**
   - 安全更新會記錄在提交中
   - 漏洞披露會負責任地處理
   - 在適當情況下發布公開安全公告

## 漏洞報告

### 我們關注的安全問題

我們歡迎以下報告：
- **程式碼漏洞**：指令碼或示例中的漏洞
- **依賴漏洞**：Python 包中的漏洞
- **加密問題**：任何程式碼示例中的密碼學問題
- **認證/授權缺陷**：檔案中的認證或授權問題
- **資料暴露風險**：配置示例中的資料洩露風險
- **注入漏洞**：SQL、命令等注入
- **SSRF / XXE / 路徑穿越** 問題

### 不在範圍內的安全問題

以下問題不在本專案範圍內：
- Claude Code 本身的漏洞（請向 Anthropic 報告）
- 外部服務或庫的問題（請向上遊專案報告）
- 社會工程學或使用者教育問題（與本指南無關）
- 沒有 PoC 的理論漏洞
- 依賴庫中已透過官方渠道報告的漏洞

## 如何報告

### 私密報告（推薦）

**如遇敏感安全問題，請使用 GitHub 的私密漏洞報告：**

1. 訪問：<https://github.com/luongnv89/claude-howto/security/advisories>
2. 點選 “Report a vulnerability”
3. 填寫漏洞細節
4. 請包含：
   - 漏洞的清晰描述
   - 受影響的元件（檔案、章節、示例）
   - 潛在影響
   - 復現步驟（如適用）
   - 建議修復方式（如果你有）

**接下來會發生什麼：**
- 我們會在 48 小時內確認收到
- 我們會調查並評估嚴重性
- 我們會與你一起修復
- 我們會協調披露時間線
- 如你願意，我們會在安全公告中致謝

### 公開報告

對於不敏感或已公開的問題：

1. **建立 GitHub Issue** 並打上 `security` 標籤
2. 包含：
   - 標題：以 `[SECURITY]` 開頭並附簡短說明
   - 詳細描述
   - 受影響的檔案或章節
   - 潛在影響
   - 建議修復方式

## 漏洞響應流程

### 評估（24 小時內）

1. 我們確認收到報告
2. 我們使用 [CVSS v3.1](https://www.first.org/cvss/v3.1/specification-document) 評估嚴重性
3. 我們判斷是否在範圍內
4. 我們向你反饋初步評估

### 開發（1-7 天）

1. 我們開發修復
2. 我們審查並測試修復
3. 我們建立安全公告
4. 我們準備釋出說明

### 披露（取決於嚴重性）

**嚴重（CVSS 9.0-10.0）**
- 立即釋出修復
- 釋出公開公告
- 提前 24 小時通知報告者

**高危（CVSS 7.0-8.9）**
- 48-72 小時內釋出修復
- 提前 5 天通知報告者
- 釋出時公開公告

**中危（CVSS 4.0-6.9）**
- 在下一次常規更新中修復
- 釋出時公開公告

**低危（CVSS 0.1-3.9）**
- 在下一次常規更新中修復
- 釋出時附帶公告

### 公佈

我們釋出的安全公告會包含：
- 漏洞描述
- 受影響元件
- 嚴重性評估（CVSS 分數）
- 修復版本
- 臨時緩解方案（如適用）
- 對報告者的致謝（如獲許可）

## 報告者最佳實踐

### 報告前

- **確認問題**：你能穩定復現嗎？
- **搜尋已有 issue**：是否已經有人報告？
- **檢視檔案**：是否已有安全使用說明？
- **測試修復**：你建議的修復是否有效？

### 報告時

- **儘量具體**：提供準確的檔案路徑和行號
- **補充上下文**：為什麼這是安全問題？
- **說明影響**：攻擊者可能做什麼？
- **提供步驟**：如何復現？
- **提出修復建議**：你會怎麼修？

### 報告後

- **保持耐心**：我們的資源有限
- **積極響應**：儘快回答後續問題
- **保持保密**：在修復前不要公開披露
- **尊重協調流程**：按時間線推進披露

## 安全頭與配置

### 倉庫安全

- **分支保護**：主分支需要 2 份批准
- **狀態檢查**：所有 CI/CD 檢查必須透過
- **CODEOWNERS**：關鍵檔案有指定審查者
- **簽名提交**：建議貢獻者使用

### 開發安全

```bash
# 安裝 pre-commit hooks
pre-commit install

# 本地執行安全掃描
bandit -c pyproject.toml -r scripts/
mypy scripts/ --ignore-missing-imports
ruff check scripts/
```

### 依賴安全

```bash
# 檢查已知漏洞
pip install safety
safety check

# 或使用 pip-audit
pip install pip-audit
pip-audit
```

## 貢獻者安全指南

### 編寫示例時

1. **永遠不要硬編碼金鑰**
   ```python
   # ❌ 不好
   api_key = "sk-1234567890"

   # ✅ 更好
   api_key = os.getenv("API_KEY")
   ```

2. **提醒安全影響**
   ```markdown
   ⚠️ **安全提示**：切勿把 `.env` 檔案提交到 git。
   請立即加入 `.gitignore`。
   ```

3. **使用安全預設值**
   - 預設啟用認證
   - 在適用場景中使用 HTTPS
   - 校驗並清理輸入
   - 使用引數化查詢

4. **記錄安全注意事項**
   - 解釋為什麼安全很重要
   - 展示安全與不安全的寫法
   - 連結權威來源
   - 明顯展示警告

### 審查貢獻時

1. **檢查是否洩露秘密**
   - 掃描常見模式（api_key=、password= 等）
   - 審查配置檔案
   - 檢查環境變數使用情況

2. **驗證安全編碼實踐**
   - 沒有硬編碼憑據
   - 正確的輸入驗證
   - 安全的認證/授權
   - 安全的檔案處理

3. **測試安全影響**
   - 這會不會被濫用？
   - 最壞情況是什麼？
   - 有沒有邊界情況？
