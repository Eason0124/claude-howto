# 面向 AI 程式碼生成的整潔程式碼規則

這些規則用於指導程式碼生成，幫助產出可維護、專業水準的程式碼。

## 有意義的命名
- 使用能體現意圖的名稱，說明它為什麼存在
- 避免誤導性命名和無意義區分，例如 `data`、`info`、`manager`
- 使用可讀、可搜尋的名稱
- 類名使用名詞，例如 `UserAccount`、`PaymentProcessor`
- 方法名使用動詞，例如 `calculateTotal`、`sendEmail`
- 避免腦內對映和編碼風格命名，例如匈牙利命名法、字首

## 函式
- 保持函式短小精煉，理想情況下少於 20 行
- 只做一件事，遵循單一職責原則
- 每個函式只處理一個抽象層級
- 限制引數數量，理想為 0-2 個，最多 3 個，避免布林標誌引數
- 不要產生副作用，函式應該做它名字所說的事
- 將命令與查詢分離，命令修改狀態，查詢返回資訊
- 優先使用異常而不是錯誤碼

## 註釋
- 程式碼本身應該儘量可自解釋，能不寫註釋就不寫
- 好的註釋包括：法律資訊、警告、TODO、公共 API 檔案
- 壞的註釋包括：重複、誤導，或者只是解釋糟糕的程式碼
- 不要註釋掉程式碼，直接刪除，版本控制會保留歷史
- 如果你需要寫註釋，先考慮重構程式碼

## 格式化
- 保持檔案小而聚焦
- 垂直格式化：相關概念放近一些，空行分隔不同概念
- 水平格式化：限制行長度在 80-120 個字元之間
- 使用一致的縮排和團隊風格
- 將相關函式分組放在一起

## 物件與資料結構
- 物件：透過抽象隱藏資料，透過方法暴露行為
- 資料結構：暴露資料，儘量少提供行為
- 迪米特法則：只和直接朋友說話，避免 `a.getB().getC().doSomething()`
- 不要毫無節制地透過 getter/setter 暴露內部結構

## 錯誤處理
- 使用異常，不要使用返回碼或錯誤標誌
- 程式碼可能失敗時，先寫 `try-catch-finally`
- 在異常資訊中提供上下文
- 不要返回 `null`，改為返回空集合或使用 Optional/Maybe
- 不要把 `null` 作為引數傳入

## 類
- 小類應按職責衡量，而不是按行數衡量
- 單一職責原則：只有一個改變原因
- 高內聚：類變數被多數方法使用
- 低耦合：類之間依賴儘量少
- 開閉原則：對擴充套件開放，對修改關閉

## 單元測試
- 快速、獨立、可重複、自我驗證、及時（F.I.R.S.T.）
- 每個測試一個斷言，或只驗證一個概念
- 測試程式碼質量應與生產程式碼質量一致
- 測試名稱要可讀，清楚描述正在測試什麼
- 遵循 Arrange-Act-Assert 模式

## 程式碼質量原則
- **DRY（Don't Repeat Yourself）**：不要重複
- **YAGNI（You Aren't Gonna Need It）**：不要為假設中的未來提前構建
- **KISS（Keep It Simple）**：避免不必要的複雜度
- **童子軍規則**：讓程式碼比你接手時更乾淨

## 應避免的程式碼壞味道
- 函式或類過長
- 重複程式碼
- 死程式碼（未使用的變數、函式、引數）
- 特性依戀（方法更關心別的類）
- 不恰當的親密關係（類彼此瞭解過多）
- 過長的引數列表
- 原始型別偏執（過度使用原始型別而不是小物件）
- switch/case 語句（考慮使用多型）
- 臨時欄位（類變數只在某些時候才使用）

## 併發
- 將併發程式碼與其他程式碼隔離
- 限制同步/加鎖資料的作用域
- 使用執行緒安全集合
- 保持同步區塊儘量小
- 瞭解你的執行模型和原語

## 系統設計
- 將構建和使用分離，例如透過依賴注入
- 複雜物件建立時使用工廠、構建器
- 面向介面程式設計，而不是面向實現程式設計
- 優先組合而不是繼承
- 設計模式只在它們簡化問題時使用，不要為了炫技而使用

## 重構
- 持續重構，不要攢成大批次一次做完
- 重構前後都要保證測試透過
- 小步前進，每次只改一處
- 常見重構包括：提取方法、重新命名、移動、內聯

## 檔案
- 自說明程式碼 > 註釋 > 外部檔案
- 公共 API 需要清晰檔案
- 檔案中要包含示例
- 檔案儘量靠近程式碼，理想情況下就在程式碼裡

---

**核心理念**：程式碼被閱讀的次數是編寫次數的 10 倍。最佳化可讀性和可維護性，而不是追求花哨。
